# MediCheck v1 Parallel Build Plan

## Goal

Turn the current MediCheck v0 recorder into a doctor workspace.

In v1, the doctor appears logged in, selects patients from a left sidebar, opens a patient session, records the patient multiple times, and gets:

- saved transcripts per recording
- model extraction results
- Tavily medication verification
- session summary
- follow-up questions for missing clinical context

The v1 loop is:

```text
Patient -> Session -> Recording -> Transcript -> Analysis -> Missing Info -> Follow-up Questions -> Next Recording
```

## Chosen Product Decisions

- Database: Supabase Postgres
- Auth: mock logged-in doctor for v1
- Doctor profile: hardcoded/seeded doctor shown in the top-right
- Sidebar: left side
- Audio storage: out of scope for v1
- Stored data: patients, sessions, transcripts, analyses, medication cards, follow-up questions
- Follow-up questions: generated by the backend with an LLM
- Existing `/triage` endpoint: keep it working for backwards compatibility

## Shared API Contracts

### `GET /workspace`

Returns the doctor and the patient/session sidebar data.

```json
{
  "doctor": {
    "id": "doctor_demo",
    "name": "Dr. Youssef",
    "avatar_url": "/doctor-avatar.jpg"
  },
  "patients": [
    {
      "id": "patient_1",
      "name": "Amal Benali",
      "age": 42,
      "sex": "female",
      "summary": "Chest pain follow-up",
      "sessions": [
        {
          "id": "session_1",
          "title": "Chest pain intake",
          "status": "active",
          "updated_at": "2026-05-16T12:00:00Z"
        }
      ]
    }
  ]
}
```

### `GET /sessions/{session_id}`

Returns the full selected session.

```json
{
  "session": {
    "id": "session_1",
    "patient_id": "patient_1",
    "title": "Chest pain intake",
    "status": "active",
    "summary": "Patient reports chest pain for three days."
  },
  "recordings": [],
  "analyses": [],
  "follow_up_questions": []
}
```

### `POST /sessions/{session_id}/recordings`

Creates a recording, analyzes it, updates session context, and returns the new state.

Request:

```json
{
  "transcript": "Patient says they have had chest pain for three days.",
  "duration_seconds": 38
}
```

Response:

```json
{
  "recording": {
    "id": "recording_1",
    "session_id": "session_1",
    "transcript": "Patient says they have had chest pain for three days.",
    "duration_seconds": 38,
    "created_at": "2026-05-16T12:10:00Z"
  },
  "analysis": {
    "pioneer": {},
    "pioneer_finetuned": {},
    "openai": {},
    "tavily_cards": [],
    "winner": "pioneer"
  },
  "session_summary": "Patient reports chest pain for three days.",
  "follow_up_questions": [
    {
      "id": "question_1",
      "question": "Can you rate the chest pain from 1 to 10?",
      "reason": "Severity has not been captured yet.",
      "priority": "high",
      "answered": false
    }
  ]
}
```

4. Update extraction results.

### B8. Add session summary generation

After each recording:

- collect all transcripts in the session
- create a short clinical summary
- save it on `sessions.summary`

Preferred implementation:

- use OpenAI if configured
- fallback to a deterministic summary based on extracted symptoms, medications, duration, and history

### B9. Add follow-up question generation

After each recording:

- collect all session transcripts
- merge all extracted entities
- identify missing clinical context
- generate 3-5 questions

Question priorities:

- `high`: red flags, allergies, contraindications, severe symptoms, missing severity
- `medium`: duration, dosage, frequency, associated symptoms
- `low`: lifestyle/context/background details

Preferred implementation:

- use OpenAI if configured
- fallback to deterministic rule-based questions

## Integration Plan

### Step 1: Agree on response contracts

Before merging either branch, confirm the frontend and backend use the same shapes from this file.

### Step 2: Backend opens endpoints first

Backend should expose:

- `GET /workspace`
- `GET /sessions/{session_id}`
- `POST /sessions/{session_id}/recordings`

These unblock most frontend integration.

### Step 3: Frontend integrates with real backend

Frontend switches from mock data to real API calls.

Keep frontend fallback demo data only for backend-unavailable states.

### Step 4: Full demo test

Test one full patient case:

1. Select patient.
2. Select active session.
3. Record first patient answer.
4. Send/analyze.
5. See extraction results and medication cards.
6. See follow-up questions.
7. Ask one follow-up question.
8. Record second patient answer.
9. Confirm session summary improves.

## Merge Strategy

Recommended order:

1. Merge backend branch into `main`.
2. Pull latest `main` into frontend branch.
3. Wire frontend to real backend.
4. Merge frontend branch into `main`.
5. Run final full-stack test.

Do not merge if:

- `/triage` is broken
- frontend build fails
- workspace endpoint is missing
- recording endpoint does not return follow-up questions
- secrets were accidentally committed

## Final Acceptance Criteria

v1 is complete when:

- doctor profile appears in the top-right
- left sidebar lists patients
- each patient shows sessions
- a session can contain multiple recordings
- each recording is saved
- each recording is analyzed
- extraction results are displayed
- Tavily medication cards are displayed
- follow-up questions are generated after each analysis
- questions can be marked answered
- session summary updates after each recording
- app works locally without real auth
- app can use Supabase when env vars are configured
